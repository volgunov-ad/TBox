# Заправки, термокоррекция и калибровка уровня топлива

Документ описывает **полный цикл** в актуальной логике приложения: как появляется запись заправки, как из неё (и из CAN) строится калибровка нелинейности датчика, как считаются **откалиброванные литры** в реальном времени и какие настройки всё это сбрасывают.

Исходная математика калибровки соответствует прототипу ветки **tochBak** (термокомпенсация + зональные коэффициенты); в приложении добавлены хранение в DataStore, привязка к поездкам и служба фона.

---

## 1. Два разных «литража»

| Поток | Откуда данные | Назначение |
|--------|-----------------|------------|
| **Процент с CAN** | Отфильтрованный `fuelLevelPercentageFiltered` | Учёт расхода поездки, эвристика «была заправка», грубая оценка литров как `% × объём бака` |
| **Откалиброванные литры** | Тот же процент + **JSON калибровки** + **температура снаружи** | Поле `fuelLevelCalibratedLiters` в репозитории и виджеты/экран данных |

Калибровка **не** подменяет сырые проценты CAN; она даёт вторую оценку объёма в литрах.

---

## 2. Когда создаётся запись заправки (`RefuelRecord`)

Условия выполняются внутри активной поездки при каждом новом отсчёте уровня (см. `BackgroundService.applyActiveTripFuelStep`):

1. Берётся текущий **отфильтрованный** уровень топлива в процентах `pctNow`.
2. С предыдущим сохранённым процентом `tripLastFuelPercent` вызывается `TripFuelAccounting.applyFuelPercentStep` (объём бака из настроек `fuelTankLiters`).

### 2.1. Правило «это заправка»

- Если за один шаг уровень **вырос не менее чем на 2 процентных пункта** (`REFUEL_RISE_PERCENT = 2`), шаг считается **заправкой**, а не расходом.
- Мелкие падения уровня (меньше **0,15** п.п.) при списании игнорируются как шум (`CONSUME_MIN_DELTA_PERCENT`).

### 2.2. Литры «по датчику» в момент заправки

При срабатывании заправки:

- `estimatedLiters` / начальные `actualLiters` = **прирост процента / 100 × объём бака** (та же линейная модель, что и для `refueledLitersThisStep` в поездке).
- В запись попадают `fuelPercentBefore` (последний уровень до скачка) и `fuelPercentAfter` (= `pctNow`).
- **`ambientTempAtRefuel`**: текущее значение **температуры снаружи** с CAN; если недоступно — **15 °C** (константа `REFUEL_AMBIENT_TEMP_DEFAULT_C`).

Дальше запись дополняется (асинхронно) координатами и ценой с АЗС при успешном запросе — это не влияет на калибровку уровня.

Заправки хранятся в репозитории (лимит **30** последних), сериализуются в JSON вместе с флагом `usedForFuelCalibration` и полем `ambientTempAtRefuel`, если оно известно.

---

## 3. Что пользователь может менять в заправке (UI)

- **Фактические литры** (`actualLiters`) — например, по чеку АЗС вместо оценки по процентам.
- **Температура при заправке** — для корректного обучения калибровки, если автоматическое значение нужно исправить.
- Остальные поля (тип топлива, цена и т.д.) — учёт стоимости и отображение.

Для **обучения** сервис использует **`actualLiters`** как «литры по чеку» и температуру из **`ambientTempForCalibrationC()`** (сохранённая температура или **15 °C**, если в записи `null`).

---

## 4. Термокомпенсация (`FuelPhysics`)

Используется линейный коэффициент объёмного расширения топлива **0,0011 на 1 °C** относительно опорной температуры **15 °C** (как в tochBak).

- **Приведение к «стандарту» (+15 °C)** — сколько бы занял тот же объём при 15 °C:

  `toStandard(liters, T) = liters / (1 + 0,0011 × (T − 15))`

- **Обратно в объём при температуре T:**

  `fromStandard(standardLiters, T) = standardLiters × (1 + 0,0011 × (T − 15))`

### 4.1. Где это применяется

| Этап | Температура | Формула |
|------|----------------|---------|
| **Обучение (`train`)** | Температура **в момент заправки** (из записи, см. выше) | Литры по чеку → `toStandard` → распределение по зонам (накопление «стандартных» литров против литров «датчика») |
| **Онлайн-оценка (`getCorrectedLiters`)** | **Текущая** температура снаружи с CAN; если `null` — **15 °C** | После зональных коэффициентов результат переводится из «стандартного» объёма в объём при текущей T через `fromStandard` |

То есть температура участвует **и** при накоплении статистики для коэффициентов, **и** при расчёте текущих откалиброванных литров на экране.

---

## 5. Модель калибровки (зоны и коэффициенты)

Классы: `CalibrationStore`, `FuelSmartEstimator`, данные `CalibrationData`.

- Бак делится на **`fuelCalibrationZoneCount`** зон (по умолчанию **5**, допустимо **3…20**). Размер зоны в «литрах датчика» = объём бака / число зон.
- Для каждой зоны накапливаются два массива одинаковой длины:
  - **`sensorLiters`** — сумма отрезков изменения «литров датчика», попавших в зону, по всем принятым заправкам;
  - **`realLiters`** — сумма соответствующих долей **стандартных** (приведённых к +15 °C) литров по чеку.
- Локальный коэффициент зоны: **K_zone = realLiters / sensorLiters** (если в зоне ещё нет данных — берётся глобальный K).
- **Глобальный K** — среднее K по «зрелым» зонам, где накоплено не меньше **5 л** датчика.
- **Уверенность (confidence)** по зоне растёт с накопленным объёмом датчика в зоне до порога **`maturityThreshold`** (литры датчика на зону; по умолчанию **80**, настраивается на вкладке «Заправки» без сброса JSON калибровки), дальше 1,0.
- **Итоговый K** для текущей точки: смесь локального и глобального по уверенности:  
  `kFinal = kLocal × confidence + kGlobal × (1 − confidence)`.

«Мёртвые зоны» датчика (низ/верх бака) задаются пропорционально объёму бака (аналог 2 л и 48 л при баке 50 л в прототипе): там уверенность в результате **дополнительно** слегка снижается.

После каждого успешного `train` снимок массивов кодируется в JSON и пишется в настройки (`fuelCalibrationJson`).

Формат JSON: объект с массивами `realLiters` и `sensorLiters` одинаковой длины (см. `FuelCalibrationJson`).

## 6. Обучение по одной заправке (`train`)

### 6.1. Запуск с UI

Пользователь инициирует обучение по выбранной заправке (кнопка на вкладке заправок). Служба получает intent **`ACTION_FUEL_CALIBRATION_TRAIN`** с extra **`refuelId`**.

### 6.2. Предусловия (иначе обучение отменяется с подсказкой)

- Заправка с таким id существует.
- **`usedForFuelCalibration` == false** (каждая запись может дать обучение только один раз).
- Заданы **`fuelPercentBefore`** и **`fuelPercentAfter`**.
- **`actualLiters` > 0**.

### 6.3. Сбор `FuelEntry`

- `litersByCheck` = `actualLiters`.
- `sensorBefore` / `sensorAfter` = проценты, переведённые в **литры линейной модели**: `% / 100 × fuelTankLiters` (тот же бак, что в настройках).
- `ambientTemp` = `ambientTempForCalibrationC()` для записи.

### 6.4. Фильтр аномалий (`FuelFilter`)

Сравнивается модуль разницы между **приростом «литров датчика»** и **литрами по чеку** с порогом **60%** от литров по чеку (`maxDeviationPercent = 0,6`).  
Учёт температуры в фильтре **не** делается (как в исходном tochBak): сравнение «в лоб».

Если заправка не прошла фильтр, JSON калибровки **не** меняется, флаг `usedForFuelCalibration` **не** ставится.

### 6.5. Шаги `train` при успехе

1. `stdLiters = toStandard(litersByCheck, ambientTemp)`.
2. По каждой зоне считается пересечение отрезка `[sensorBefore, sensorAfter]` с зоной; доля `overlapShare` от полного `sensorAfter − sensorBefore` умножается на `stdLiters` и добавляется в `addData(zone, stdLiters × overlapShare, длина пересечения в литрах датчика)`.
3. Сохранение JSON через callback → DataStore.
4. У записи заправки выставляется **`usedForFuelCalibration = true`** и список заправок персистится.

## 7. Непрерывный расчёт откалиброванных литров

В `BackgroundService` запущен корутинный коллектор (`startFuelCalibratedLitersWatcher`), который при изменении любого из:

- объём бака,
- число зон,
- строка JSON калибровки,
- отфильтрованный процент топлива,
- температура снаружи,

пересобирает при необходимости `FuelSmartEstimator` и вызывает:

`getCorrectedLiters(sensorLiters, currentTemp)`,

где `sensorLiters = pct / 100 × tank`, `currentTemp` — наружная температура или **15 °C**.

Результат публикуется в `CanDataRepository`:

- `fuelLevelCalibratedLiters` — литры;
- `fuelCalibrationConfidence` — число 0…1 (с учётом зоны и «мёртвых» зон).

## 8. Настройки и сбросы калибровки

| Действие | Что происходит с JSON калибровки | Флаги заправок |
|----------|-----------------------------------|----------------|
| Сохранён новый **объём бака** (с подтверждением сброса калибровки) | JSON **очищается** | Все `usedForFuelCalibration` **сбрасываются** в `false` |
| Сохранено новое **число зон** (с подтверждением) | JSON **очищается** | Флаги заправок **сбрасываются** |
| Кнопка **«Сбросить калибровку»** (только модель) | JSON **очищается** | Флаги заправок **сбрасываются** |
| Обычное сохранение JSON после `train` | Обновляется | — |
| Сохранение **порога зрелости зоны** (`maturityThreshold`) | **Не** меняется | **Не** меняются |

Число зон и объём бака в UI, влияющие на сброс, применяются через отдельные действия ViewModel (не «на каждый символ»). Порог зрелости сохраняется отдельно и только меняет смешивание локального и глобального K при том же накопленном JSON.

После сброса те же заправки можно снова использовать для обучения (флаг снят), если данные подходят под фильтр.

---

## 9. Краткая цепочка «от CAN до экрана»

1. CAN → буфер → **`fuelLevelPercentageFiltered`**.
2. При росте ≥ 2 п.п. во время поездки → **`RefuelRecord`** (+ поездка: расход не увеличивается на этот скачок, учитывается объём заправки).
3. Пользователь при необходимости правит литры/температуру.
4. По кнопке → **`train`** (фильтр → зоны → JSON).
5. Постоянно → **`getCorrectedLiters`** → **откалиброванные литры** и уверенность.
