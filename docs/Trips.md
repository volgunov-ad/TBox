# Поездки: как это работает (простыми словами)

Ниже описано, как устроены **автоматические поездки** в приложении, в актуальной логике.

## Что такое поездка

Поездка — это запись с началом, (обычно) концом, пробегом и тремя независимыми типами времени:

- **время в движении** (`moving`) — двигатель работает, скорость > 0;
- **время в простоях** (`idle`) — двигатель работает, скорость = 0;
- **время парковки** (`parking`) — двигатель выключен, RPM = 0.

Запись живёт в памяти и периодически сохраняется в хранилище.

## Когда поездка **начинается**

- Двигатель завели (RPM стало > 0), активной поездки нет, и есть корректный переход через ноль RPM — создаётся **новая** поездка.
- Если при первом запуске службы двигатель уже работает и в хранилище нет активной поездки — поездка тоже может стартовать.
- Если в хранилище уже есть активная поездка (служба перезапустилась в процессе) — она продолжается.

## Когда поездка **заканчивается** (автоматически)

- Переход RPM: **было > 0, стало 0** — поездка закрывается (фиксируется `endTime`).
- Далее действует **интервал разделения поездок** (`splitTripTime`, N минут).

## Короткая остановка (в пределах `splitTripTime`)

- Если двигатель снова запустили раньше, чем прошло N минут, продолжается **та же** поездка.
- Время между `RPM -> 0` и следующим `RPM > 0` добавляется в **время парковки**.
- Это же правило работает и если за паузу ГУ/служба перезапускались: при восстановлении поездки в пределах окна split такой отрезок тоже попадёт в `parking`.

## Длинная стоянка (дольше `splitTripTime`)

- Если прошло больше N минут, следующая поездка при запуске двигателя будет **новой**.
- Накопленное за длинную паузу «парковочное» время никуда не переносится:
  - не добавляется в старую поездку;
  - новая начинается с нуля.

## Как считаются времена внутри поездки

Пока поездка активна и RPM > 0:

- скорость > 0 -> добавляется `moving`;
- скорость = 0 -> добавляется `idle`.

Пока RPM = 0:

- `moving` и `idle` не растут;
- соответствующие паузы учитываются как `parking` только в сценариях продолжения той же поездки (см. окно split).

## Холодное восстановление и выключение ГУ без запуска двигателя

Если служба восстановила недавно завершённую поездку (в окне split), но в этой сессии двигатель так и не был запущен:

- при выключении ГУ возвращается старый `endTime`;
- откатываются только временные изменения, добавленные из-за такого восстановления.

Это нужно, чтобы один и тот же парковочный отрезок не посчитался дважды при следующем включении ГУ.

## Формулы метрик

- **Общее время** = `moving + idle + parking`.
- **Средняя скорость без простоев** = `distance / moving`.
- **Средняя скорость поездки** = `distance / (moving + idle)`  
  (парковка из средней скорости исключена).
- **Расход топлива (л/100км)** = `fuel_used * 100 / distance` (если distance > 0).

## Где что отображается

- **Меню Поездки**: показывает `moving`, `idle`, `parking`, `total`, скорости, расход и другие поля.
- **Экспорт поездок**: включает `Время парковки`.
- **Виджет Поездка (полный)**: показывает `Время парковки`.
- **Виджет Поездка (упрощённый)**: `Время парковки` не показывает.
- **Виджет Поездка (мини)**: показывает только пробег, общее время и расход.

## Обратная совместимость

- У уже сохранённых старых поездок `parkingTime` отсутствует в JSON и читается как `0`.
- Миграция старых данных не выполняется.

## Ручное «завершить поездку» в интерфейсе

- Текущая поездка закрывается текущим временем.
- Сразу создаётся новая (с учётом текущего RPM).

## Импорт/перезагрузка поездок из бэкапа без полного перезапуска

- Данные из импорта принимаются как источник истины.
- Автоматическое «переоткрытие недавно завершённой поездки» при такой перезагрузке не запускается.

## Краевые случаи

- Если часы ГУ ушли назад (текущее время меньше `endTime`), недавно завершённую поездку не продолжают.
- Ложные RPM по CAN могут имитировать запуск двигателя и влиять на авторазделение поездок.
- Хранится ограниченное число последних поездок; старые вытесняются.
